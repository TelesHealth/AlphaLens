(reading 'amountOverride')" when clicking
Approve on a pending order.

find POST /api/trading/pending/:id/approve.

The route is trying to read amountOverride from
the pending order but the field doesn't exist
or is undefined.

Fix:
1. Find where amountOverride is read from the
   pending order object.
2. Add a safe fallback:
   const amountUsd = order.amountOverride
     ?? order.amountUsd
     ?? 50;
   Use amountUsd for the trade execution.

3. If amountOverride column doesn't exist in
   the pending_orders schema in lib/db/src/schema/,
   remove all references to it and use amountUsd
   consistently throughout the approve flow.

4. Check the entire approve route for any other
   undefined field reads and add null guards.

Run pnpm run typecheck. Zero errors required.

BUG #28 — P1 — Inconsistent platform status
across pages

Kalshi shows "configured" in Settings and
Trading Overview but "not configured" in
Trading Pending tab and Briefing cards.

Root cause: Multiple places check platform
status using different methods:
- Some read from user_trading_accounts table
  (correct — per-user)
- Some read from process.env.KALSHI_EMAIL
  (wrong — this is the admin fallback only)

Fix in two places:

1. In artifacts/api-server/src/routes/trading.ts,
   find GET /api/trading/accounts.
   This endpoint must ALWAYS check the current
   user's credentials from user_trading_accounts
   first, before falling back to env vars.
   The response must reflect the logged-in user's
   configuration, not the admin env vars.

2. In artifacts/alpha-lens/src/pages/briefing.tsx,
   find where the Execute Live Trade button checks
   if a platform is configured.
   It must call GET /api/trading/accounts and
   check the response — not hardcode any platform
   status or check env vars directly.

3. In artifacts/alpha-lens/src/pages/trading.tsx,
   the Pending tab must use the same
   GET /api/trading/accounts call to determine
   platform status — not an independent check.

Single source of truth:
GET /api/trading/accounts always returns the
current user's platform status. All UI components
must use this endpoint. No component should
independently check platform configuration.

Run pnpm run typecheck. Zero errors required.

Verify:
- Log in as Charlize
- Go to Settings — Kalshi shows Configured
- Go to Trading Overview — Kalshi shows Configured
- Go to Briefing — Execute Live Trade button
  shows "Execute via KALSHI" not disabled state
- Go to Trading Pending — platform shown correctly
